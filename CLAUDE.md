# AUVORA — Project Instructions for Claude Code

## What AUVORA is
AUVORA is a mobile-first premium “aura operating system.”
It is not a generic AI app, not a dashboard, and not a simple vibe generator.

AUVORA should feel:
- editorial
- cinematic
- sensual
- fluid
- atmospheric
- intelligent
- premium
- taste-led
- emotionally rich

AUVORA should not feel:
- cramped
- generic
- over-carded
- dashboard-like
- container-heavy
- plain
- AI-made
- cold
- form-first
- samey across sections

## Core quality bar
Every major screen must have:
- a clear hero moment
- strong hierarchy
- breathing room
- meaningful section identity
- premium motion
- believable utility under the polish

Default to:
- fewer boxes, bigger moments
- editorial composition over dashboard composition
- detail pages richer than overview cards
- atmosphere without sacrificing usability
- clear, intentional, premium mobile-first layouts

Do not default to:
- repeated stacked cards
- chip overload
- cramped rows of controls
- generic gradients
- same-template screens with relabeled content
- luxury styling pasted onto ordinary SaaS structure

## Critical product direction
### Style / wardrobe
Do not treat style as “generate an outfit the user probably does not own.”

Style should behave like a Pinterest-grade inspiration engine:
- visual direction
- silhouette guidance
- palette direction
- moodboards
- occasion-based styling inspiration
- aesthetic references
- “dress the aura” direction
- “shop your closet” only when real wardrobe data exists

Inspiration is more important than prescription.

### Scent
Scent should feel like a premium fragrance concierge.

Make fragrance intelligence visible through:
- richer fragrance detail pages
- stronger scent identity/profile views
- clearer mood/use-case guidance
- more intelligent layering guidance
- expressive storytelling around signature, reach, and wear context

Concierge intelligence is more important than plain catalog utility.

### Whole product
AUVORA should feel like a world, not a tool.
Each major section should feel meaningfully distinct in:
- tone
- pacing
- layout
- composition
- density
- interaction style
- atmosphere

No relabeled template screens.

## Working method
Before major implementation work:
1. Diagnose what is weak in the current design
2. Identify what feels generic, cramped, flat, or under-directed
3. Propose a stronger route
4. Remove weak ideas before adding new ones
5. Then implement cleanly

Think system-first before screen-first.
Think screen-first before component-first.
Think experience before decoration.

Do not preserve weak assumptions just because they already exist.

## UX / UI principles
- Editorial before dashboard
- Fewer boxes, bigger moments
- Every major screen needs a focal point
- Inputs should feel like invitation, not form filling
- Motion should communicate feeling and hierarchy
- Detail views should feel immersive, not just expanded cards
- Reduce density before adding features
- Use visual restraint with confidence
- Make the app feel alive, not busy
- Premium means clarity + taste + control, not ornament alone

## Motion direction
Motion should feel:
- calm
- fluid
- layered
- sensual
- premium
- cinematic
- responsive

Prefer:
- layered reveals
- soft scale/blur transitions
- restrained spring motion
- elegant page and detail transitions
- subtle ambient movement
- purposeful loading states

Avoid:
- random floating effects
- excessive bounce
- gimmicky motion
- overly busy transition choreography
- motion that slows core usage

## Visual direction
Use:
- strong typography hierarchy
- generous spacing
- dark luxury styling with restraint
- controlled glow/blur/material treatment
- cleaner, larger compositional moments
- differentiated section identities
- premium contrast and readability

Reduce:
- pill overload
- repetitive cards
- overuse of outlines
- cramped text stacking
- nested containers
- visual sameness
- micro-sections competing for attention

## Engineering direction
- Mobile-first always
- Prefer TypeScript
- Keep code modular, reusable, and production-ready
- Use clean file structure and scalable component boundaries
- Avoid fragile hacks unless clearly necessary
- Preserve performance while improving visual richness
- Build realistically for a real Next.js app
- Do not overcomplicate architecture for aesthetic reasons

## Use available skills and libraries intelligently
Assume high-quality Claude Code skills and libraries are available.
Use them intentionally, not decoratively.

For UI/design work:
- use installed design skills to avoid generic AI-made interfaces
- push for better hierarchy, rhythm, spacing, and motion
- refine weak first passes instead of accepting them

For motion/interaction work:
- favor premium, controlled, emotionally intelligent motion
- use installed motion/design capabilities to improve feel, not add noise

For engineering work:
- use cleanup/refactor/debug skills when appropriate
- prefer strong reusable patterns over rushed patches
- iterate when a result is technically correct but not premium enough

If premium UI/component/design libraries are available and materially improve quality, use them intelligently instead of reinventing weaker versions from scratch.

## Failure modes to avoid
Avoid outputs that feel like:
- generic AI app UI
- dashboard with luxury colors
- repeated card stacks everywhere
- too many pills/chips
- same structure across all tabs
- beautiful but empty
- premium but unusable
- feature-heavy but emotionally flat
- polished but forgettable

## What success looks like
AUVORA should surpass Scentory in:
- visual quality
- emotional richness
- motion language
- UX clarity
- section distinctiveness
- product coherence
- perceived intelligence
- overall premium feel

When there is tension between:
- more features vs better atmosphere
- more density vs better clarity
- safer UI vs more memorable UI

choose the route that makes AUVORA feel more authored, more premium, and more category-defining.

## Response behavior
When responding on this project:
- be decisive
- challenge weak product assumptions
- explain the better route briefly
- then implement clearly
- do not settle for generic first-pass design
- favor strong taste and real usability together

If the first solution still feels cramped, flat, or templated, improve it before finalizing.
